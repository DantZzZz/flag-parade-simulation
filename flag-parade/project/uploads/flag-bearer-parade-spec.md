# Flag Bearer Parade — Creative Coding Project Spec

## Project Overview

An interactive 3D flag bearer parade simulator where the user acts as a **director**, commanding stylized humanoid flag bearers to march between formations on a parade ground. The project emphasizes **beautiful visuals and cinematic presentation**, with full camera control and choreography tools.

Built with **React + React Three Fiber (R3F)** for declarative 3D rendering with a React-based UI layer.

## Visual Design

### Aesthetic Direction

The project prioritizes **beauty first**. The overall feel should be elegant, cinematic, and satisfying to watch — closer to a visual art piece than a game engine demo.

### Environment / Mood Presets (v1: 1 mood)

**Minimal / Abstract (v1)** — Dark void background with a subtle grid on the ground plane. Clean, no distractions. Focus entirely on the bearers and their flags. Soft ambient lighting with gentle shadows.

**v2 Moods:**

- **Golden Hour** — Warm directional sunlight casting long shadows. Soft orange-pink sky. Ground is a matte parade surface with subtle texture. Atmospheric haze/fog near the horizon.
- **Night with Dramatic Spotlights** — Dark environment. Multiple colored spotlights from above illuminating the formation. Volumetric light cones if performant. Ground reflects light subtly. Stars or dark gradient sky.
- **Bright Daylight** — Classic parade ground in full daylight. Clear blue sky, even lighting, natural shadows.

### Flag Bearers — Stylized Minimal Humanoids

- Simple geometric body: capsule or rounded cylinder torso, sphere head, thin cylinder limbs
- No face or fine detail — think Monument Valley / Alto's Adventure aesthetic
- Uniform color scheme (e.g. dark navy body, white/gold accents)
- Each bearer holds a vertical flagpole with a cloth flag

### Flag Behavior (Detailed)

The flags are a primary visual element and must feel **natural and alive**.

#### Wind Simulation

- A **global wind direction and strength** drives all flags. Wind is a constant directional force (e.g. blowing from the left) that causes flags to billow in that direction.
- The wave motion is generated by a **vertex shader** applying layered sine waves along the flag's horizontal and vertical axes, with the wind direction determining the primary wave propagation direction.
- **Per-bearer variation**: Each flag has a small random phase offset (fixed at spawn) so they wave in unison but not identically — like real flags in the same wind. The variation should be subtle (±5-10% of the wave cycle), enough to break the robotic feel without looking chaotic.
- **Wind parameters** (exposed in UI): direction (angle), strength (affects wave amplitude and how far the flag extends from the pole).

#### Movement Response & Inertia

- When a bearer **changes direction or starts/stops moving**, the flag **trails behind with inertia**. The flag's orientation lags the bearer's facing direction, then gradually settles to align with the wind.
- Implementation: track each bearer's velocity vector. The flag's rest angle is a blend of the global wind direction and the inverse of the bearer's movement direction (drag). When the bearer stops, the flag eases back to pure wind-driven behavior over ~1-2 seconds.
- During **direction changes and acceleration**, the flag wave **increases in turbulence** — higher frequency ripples and slightly larger amplitude, as if the cloth is being buffeted by the change in airflow. This settles back to the calm wind-wave over ~1 second after the bearer reaches steady state (constant velocity or stationary).

#### Flag Appearance & Configuration

- **v1: Pattern presets** — The user selects from a set of flag pattern presets, each recolorable:
  - **Solid** — single color fill
  - **Horizontal Stripe** — two colors, top/bottom split (configurable split ratio: 50/50, 33/66, etc.)
  - **Vertical Stripe** — two colors, left/right split
  - **Diagonal Stripe** — two colors, diagonal division
  - **Center Circle Emblem** — solid background with a circle of a second color centered on the flag
  - **Cross** — solid background with a cross stripe in a second color
  - **Border** — solid fill with a contrasting border/edge stripe
- Each preset exposes a **primary color** and **secondary color** picker.
- All flags in the parade use the same pattern and colors (uniform look).
- **v2: Freeform flag designer** — position stripes, resize emblems, pick shapes, upload custom SVG patterns.

#### Flag Shader Implementation Notes

- The flag is a **subdivided plane geometry** (e.g. 20×12 segments) attached to the pole along one edge.
- The vertex shader displaces vertices using: `displacement = sin(position.x * freq1 + time * speed1 + phaseOffset) * amplitude1 + sin(position.y * freq2 + time * speed2) * amplitude2`
- Vertices closer to the pole (x=0) have displacement clamped to zero (flag is attached). Displacement scales with distance from pole.
- Wind direction rotates the displacement vector. Bearer velocity adds a secondary displacement component.
- Turbulence multiplier: a uniform float `uTurbulence` that scales frequency and amplitude, driven by the bearer's acceleration magnitude with exponential decay back to 1.0.

### Bearer Behavior (Detailed)

#### Marching Cadence — "Well-Rehearsed Unison"

- All bearers share a **single global tempo** (a master clock / metronome). The tempo drives the marching cycle: leg alternation, arm swing, torso bob.
- Each bearer has a **tiny fixed random phase offset** (assigned at spawn, ±2-5% of the full step cycle). This means every bearer steps at the same *rate* but is shifted by a few milliseconds — like a real drill team that's trained hard but isn't mechanically perfect.
- The result: from a distance they look perfectly synchronized; up close you can see the slight organic stagger. This is critical to the "natural but disciplined" feel.

#### Idle State — Pacing in Place

- When bearers are **not transitioning** (holding a formation), their default idle state is **marching in place** — feet rise and fall in the marching rhythm, arms swing subtly, but the bearer does not travel.
- This keeps the scene alive and energetic even when no formation change is happening.
- The same global tempo + phase offset system drives the idle pacing, so it stays consistent with the traveling march.

#### Movement Speed

- Bearers move at a **base marching speed** that is configurable via the UI (a speed slider).
- The speed model is **ease-in-out** with very slight per-bearer variation:
  - When a formation transition starts, bearers accelerate from idle to marching speed (ease-in over ~0.5s).
  - They travel at a steady speed (the configured base speed ± a tiny per-bearer variation, ~±2-3%).
  - As they approach their target, they decelerate (ease-out over ~0.5-1s) and settle into pacing-in-place at the target position.
- The per-bearer speed variation means they don't all arrive at exactly the same instant, which looks natural — the formation "crystallizes" over a brief moment rather than snapping into place.

#### Procedural Animation Details

- **Legs**: alternating forward/back rotation on the hip joint, sine-wave driven. Amplitude and frequency tied to the global tempo.
- **Arms**: opposite-phase to legs (left arm forward when right leg forward). Subtler amplitude than legs. The flag-holding arm has reduced swing (it's bracing the pole).
- **Torso**: very subtle vertical bob (up on mid-stride, down on foot-plant), ~1-2% of bearer height.
- **Head**: essentially fixed, very slight forward tilt. No head bob.
- All driven by: `phase = globalTime * tempo + bearerPhaseOffset`

### Ground Plane

- Large flat surface beneath the bearers
- Visual treatment: subtle grid pattern on a dark surface (matching minimal/abstract mood)
- Should extend beyond the formation area to give a sense of space
- Optional: subtle shadow receiving

## Core Features

### 1. Formation System

#### Preset Formations

The app ships with a library of built-in formations. Each formation is a function that takes `n` (number of bearers) and returns an array of `{ x, z }` target positions (y is always ground level).

**Required presets:**

- **Grid** — rectangular rows and columns, configurable rows × cols
- **Circle** — bearers evenly spaced on a circle, configurable radius
- **Concentric Circles** — multiple rings, configurable ring count
- **V-Shape / Chevron** — classic parade wedge, configurable angle
- **Diamond** — rotated square
- **Line / Single File** — one row or one column
- **Cross / Plus** — two intersecting lines
- **Star** — 5-pointed star outline
- **Arrow** — forward-pointing arrow shape
- **Spiral** — Archimedean spiral from center outward
- **Heart** — heart shape outline
- **Square Outline** — hollow square perimeter
- **Staggered Rows** — offset grid (like bricks)
- **Random Scatter** — random positions within a bounding area (for contrast/chaos)

#### Custom Formations via Mini-Map

- A **2D top-down mini-map panel** (overlay or side panel) shows the current formation from above
- The user can **click to place points** on the mini-map, each point representing a bearer's target position
- Points are draggable for fine-tuning
- The mini-map shows a scale grid for spatial reference
- A "distribute" button can auto-space selected points evenly along a drawn path
- Custom formations are saved to a local list and can be named

#### Formation Transition Engine

- When a new formation is selected, each bearer is assigned a target position using an optimal assignment algorithm (minimize total travel distance — use the Hungarian algorithm or a greedy nearest-neighbor heuristic for performance at scale)
- Bearers march to their new positions simultaneously
- Transition duration is configurable (default: ~4-6 seconds depending on distance)
- Easing: smooth ease-in-out on the path, not linear
- Bearers should avoid collisions during transition — implement basic steering/separation behavior so they don't walk through each other

### 2. Bearer Count Control

- Default: **200 bearers** (large parade feel)
- Adjustable via a slider: range **10 – 500**
- When count changes:
  - If increased: new bearers march in from the edge of the field
  - If decreased: excess bearers march off the field edge and despawn
- Performance target: maintain **60fps at 300 bearers** on a mid-range GPU. At 500, graceful degradation (lower flag cloth detail, simpler shadows) is acceptable.
- Use **instanced meshes** (`THREE.InstancedMesh`) for the bearer bodies to keep draw calls low

### 3. Camera System

The user is the "director" and has full camera control. Three camera modes:

#### a) Orbit Mode (Default)

- Smooth orbit around the center of the formation
- Mouse drag to rotate, scroll to zoom, right-drag to pan
- Uses `@react-three/drei`'s `OrbitControls` with damping enabled
- Auto-rotate option: slowly orbits when idle

#### b) Camera Presets

Instant (with smooth transition) jumps to predefined camera angles:

- **Aerial / Bird's Eye** — directly above looking down
- **Ground Level Tracking** — low camera at bearer height, looking across the formation
- **Hero Shot** — slightly elevated, looking at the formation from the front at ~30°
- **Close-Up** — zoomed into a few bearers, shallow depth-of-field effect (DOF post-processing)
- **Wide Establishing** — far back, captures the entire formation with surroundings
- **Diagonal Fly-Over** — elevated angle from a corner

#### c) Free-Fly Mode

- WASD + mouse look (FPS-style controls)
- Shift to move faster, scroll to adjust speed
- Good for finding unique cinematic angles manually
- A "save camera position" button lets the user bookmark custom angles

#### d) Choreographed Camera Paths (v1)

- The user can **record a camera path**: press record, fly the camera around, press stop. The path (position + lookAt over time) is saved as keyframes.
- Camera paths can be **played back** during a choreography sequence
- Basic keyframe editing: the user can see keyframe markers on a mini-timeline and delete or adjust them
- Camera paths are stored alongside formation sequences in the choreography data

### 4. Choreography / Sequence System

The workflow is **step-through** style (not a full timeline editor):

1. User selects a formation → bearers transition to it
2. User watches and adjusts (camera angle, timing)
3. User clicks "Add to Sequence" → current formation + camera state + transition duration is appended to a sequence list
4. User selects the next formation → repeat
5. The sequence list is displayed as an ordered list in a panel (reorderable via drag-and-drop)
6. "Play Sequence" plays back all formations in order with their saved camera movements and transition timings
7. Each step in the sequence stores:
   - Formation ID (preset name or custom formation reference)
   - Bearer count (if changed)
   - Transition duration (seconds)
   - Hold duration (how long to stay in this formation before transitioning to the next)
   - Camera state (position, target, or camera path reference)
   - Mood preset (if changed between steps)

### 5. Export & Sharing

#### Video Export

- Uses the browser's `MediaRecorder` API to capture the WebGL canvas stream
- User clicks "Record", plays a sequence (or manually directs), clicks "Stop"
- Outputs a `.webm` file (or `.mp4` if using a compatible encoder)
- Resolution options: 720p, 1080p, 4K
- Option to record at a fixed framerate (e.g., 30fps or 60fps) for smooth output regardless of real-time performance

#### Shareable URL

- The entire choreography state (sequence of formations, camera states, bearer count, mood) is serialized to JSON
- JSON is compressed (e.g., `pako` / deflate) and base64-encoded into a URL hash fragment
- Opening the URL loads the choreography and auto-plays the sequence
- For complex choreographies that exceed URL length limits, fall back to a "copy JSON to clipboard" option

### Right Sidebar — Director Controls

A single collapsible sidebar on the right side of the screen, divided into **accordion sections** that expand/collapse. Only one or two sections open at a time to save space. The sidebar is translucent/glassmorphism, scrollable, and can be fully hidden with the `H` key.

#### Section 1: Formation Presets

- Grid of thumbnail icons showing each preset formation as a small top-down dot pattern
- Clicking a preset triggers the formation transition immediately
- Active formation is highlighted
- "Add to Sequence" button pinned at the bottom of this section

#### Section 2: Custom Formation (Mini-Map)

- A 2D top-down interactive canvas showing the current bearer positions as dots on a grid
- **Click** to place a new point, **drag** to reposition, **right-click** to delete
- Grid snapping toggle for precise placement
- "Distribute evenly" button to auto-space selected points along a path
- "Apply" button sends the custom formation to the 3D scene
- "Save" button stores the custom formation with a user-defined name
- Saved custom formations appear as thumbnails alongside the presets in Section 1

#### Section 3: Bearers

- **Count**: Slider (range 10–500, default 200) with numeric input field
- **Speed**: Slider controlling the base marching speed (affects both traveling and pacing-in-place tempo). Labeled with descriptive values (e.g. "Slow March" → "Standard" → "Quick March")

#### Section 4: Flags

- **Pattern**: Dropdown or visual grid of pattern presets (Solid, Horizontal Stripe, Vertical Stripe, Diagonal Stripe, Center Circle, Cross, Border)
- **Primary Color**: Color picker
- **Secondary Color**: Color picker (disabled/hidden when pattern is "Solid")
- **Wind Direction**: Circular angle picker (drag a handle around a circle to set wind angle) with a small arrow indicator showing direction
- **Wind Strength**: Slider (calm → moderate → strong), affects flag wave amplitude and extension

#### Section 5: Plaza

Lighting and environment controls for the minimal/abstract scene.

**Recommended v1 controls:**

- **Ambient Light Intensity**: Slider (dim → bright). Controls the overall base illumination of the scene.
- **Grid Visibility**: Toggle on/off + opacity slider. Controls the ground grid pattern.
- **Ground Reflectivity**: Slider (matte → reflective). Controls how much the dark ground reflects the bearers and spotlights.
- **Formation Spotlight**: Toggle on/off. A wide overhead spot (or cluster) that illuminates the bounding area of the current formation. Follows the formation when it changes. Configurable:
  - Color picker
  - Intensity slider
  - Softness slider (hard-edged pool of light → soft gradient falloff)
- **Accent Spotlights**: Toggle on/off. Additional dramatic spots with two behavior modes (user toggles between them):
  - **Distributed** — 3-5 spots auto-positioned at key points of the formation (front, center, wings/edges). They reposition when the formation changes.
  - **Roaming** — A single (or pair) of spots that slowly sweep across the formation in a smooth path, creating a cinematic scanning effect.
  - Configurable: color picker, intensity slider, count (for distributed mode), sweep speed (for roaming mode).

#### Section 6: Camera

- **Mode toggle**: Orbit / Presets / Free-Fly
- **Camera Presets**: Grid of labeled buttons (Aerial, Ground Level, Hero Shot, Close-Up, Wide, Diagonal). Visible in all modes — clicking one smoothly transitions to that angle.
- **Auto-Rotate**: Toggle on/off (orbit mode only) + speed slider
- **Record Camera Path**: Start/Stop recording button. When recording, a red indicator pulses. The recorded path appears as a named entry in a list below.
- **Saved Camera Paths**: List of recorded paths with Play / Delete buttons
- **Save Current Position**: Button to bookmark the current camera angle with a name

### Bottom Bar — Sequence & Export

- Horizontal strip across the bottom of the screen, translucent
- **Sequence Steps**: Ordered cards displayed left to right, each showing a formation thumbnail icon and duration. Drag-and-drop to reorder. Click to jump to that step.
- **Playback Controls**: ◀ (previous step), ▶ Play / ⏸ Pause, ⏹ Stop, ▶| (next step)
- **Record Button**: ⏺ Starts/stops video recording. Red pulsing indicator when active. Click opens a small popover to select resolution (720p / 1080p / 4K) before recording starts.
- **Share Button**: 🔗 Generates a shareable URL (or copies JSON to clipboard if too long). Shows a confirmation toast.

### Design Principles for UI

- **Translucent / glassmorphism** sidebar and bottom bar — the 3D scene should always feel primary
- Dark theme to complement the minimal/abstract aesthetic
- Minimal text, icon-driven where possible
- Panels can be collapsed/hidden for a clean full-screen view (toggle with `H` key)
- Responsive: panels stack or collapse on smaller viewports but primary target is desktop

## Performance Considerations

- **Instanced Rendering**: All bearer bodies use `InstancedMesh`. The flag cloth can be a shared geometry with per-instance vertex shader offsets (phase offset per bearer for natural variation).
- **LOD**: At high bearer counts (300+), reduce flag cloth complexity (fewer wave vertices), simplify shadow maps, reduce marching animation bone count.
- **Frustum Culling**: Three.js handles this by default for instanced meshes.
- **Shadow Optimization**: Formation spotlight uses a single shadow map. Accent spotlights (distributed mode, up to 5) should have shadows disabled or use a low-res shared shadow map. Roaming spotlight can have its own shadow map since it's only 1-2 lights.
- **Animation**: The marching cycle should be **procedural** (sine-wave based limb positions), not skeletal — this avoids per-bearer `AnimationMixer` overhead.
- **Transition Pathfinding**: The assignment algorithm runs once per formation change on the main thread. For 500 bearers, a greedy nearest-neighbor assignment should complete in <16ms. If not, move to a Web Worker.

## v2 Roadmap (Out of Scope for v1)

- Golden hour, night with spotlights, and bright daylight mood presets (with mood switching UI and smooth light transitions)
- Freeform flag designer (position stripes, resize emblems, pick shapes, upload custom SVG patterns)
- Timeline-based choreography editor (visual track with drag-to-resize durations)
- AI-assisted formation creation (type a description → generate positions)
- SVG/image import for formation shapes
- Multiple flag groups (assign different flags to subgroups of bearers)
- Sound system (marching drums, ambient audio, fanfare triggers)
- Formation morphing (continuous interpolation between two formations, not step-based)
- Particle effects (confetti, sparks on formation completion)
- Multi-formation simultaneous display (split bearers into independent groups)
- Mobile / touch support
- Save choreographies to cloud storage / user accounts

## Key Architectural Decisions & Rationale

1. **React Three Fiber over raw Three.js** — The project has significant UI needs (panels, mini-map, sequence editor). R3F lets us use React for both the UI and the 3D scene declaratively, sharing state via Zustand without bridging two frameworks.

2. **Instanced Mesh over individual meshes** — At 200-500 bearers, individual `Mesh` objects would create hundreds of draw calls. `InstancedMesh` renders all bearers in 1-2 draw calls. The marching animation is applied by updating the instance matrix each frame.

3. **Procedural animation over skeletal** — Skeletal animation (bones + skinning) per bearer would be expensive at scale. A procedural approach (computing limb positions from sine functions each frame and writing to instance transforms) is O(n) simple math with no animation mixer overhead.

4. **Greedy assignment over Hungarian algorithm** — The Hungarian algorithm is O(n³) which is too slow for 500 bearers in real-time. A greedy nearest-neighbor pass (sort by distance, assign closest available) runs in O(n²) worst case and produces visually convincing results even if not mathematically optimal.

5. **URL hash sharing over backend** — Keeps the project fully client-side with no server dependency. Choreographies with up to ~20 steps typically fit within URL length limits when compressed. Clipboard fallback handles edge cases.

6. **Step-through over timeline editor** — A full timeline editor is a complex UI project on its own. Step-through is intuitive (pick → watch → next), achieves the choreography goal, and can be upgraded to a timeline in v2 without architectural changes.
